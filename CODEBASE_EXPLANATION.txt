═══════════════════════════════════════════════════════════════════════════════
PROJECT OVERVIEW
═══════════════════════════════════════════════════════════════════════════════

Nightmare Library is a digital library application for EPUB and PDF books with:
- Obsidian-black theme UI
- Deployed on Cloudflare Pages (GitHub integration)
- 10 Cloudflare D1 databases for 5GB total storage capacity
- 10 cascading storage providers for file backup and redundancy
- Session-based authentication with KV storage
- Text-only and PDF readers with progress tracking

KEY ARCHITECTURE COMPONENTS
═══════════════════════════════════════════════════════════════════════════════

1. DATABASE SHARDING (functions/db-router.ts)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Problem: Cloudflare D1 has 500MB limit per database
Solution: Distribute across 10 D1 databases (5GB total capacity)

Implementation:
- hashBookId(): Consistent hash function converts book ID to database index 0-9
- DatabaseRouter class: Manages all 10 database instances
- createDatabaseRouter(env): Factory function to initialize router with DB_1-DB_10
- queryAll(): Queries all 10 databases and aggregates results
- queryForBook(bookId): Routes single-book queries to correct database

Example flow:
  Book ID "book-123" → Hash to 5 → Query DB_5 → Return result

2. STORAGE CASCADING (functions/storage-proxy.ts)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Problem: Want redundancy + avoid vendor lock-in
Solution: Try 10 storage providers in order, fail to next if error

Provider Priority Order:
  1. Google Drive (15GB free)
  2. Dropbox (2GB free)
  3. OneDrive (5GB free)
  4. pCloud (10GB free)
  5. Box (10GB free)
  6. Yandex Disk (10GB free)
  7. Koofr (10GB free)
  8. Backblaze B2 (10GB free)
  9. Mega.nz (20GB free)
  10. GitHub (4GB free - final fallback)

File upload process:
  Upload file → Try GDrive → If fails, try Dropbox → ... → If all fail, error

3. AUTHENTICATION & SESSIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━
Mechanism:
- Sessions stored in Cloudflare KV (KV_SESSIONS namespace)
- Middleware validates session cookie on every request
- Sessions include user ID, expiration, IP, User-Agent for hijack protection
- JWT_SECRET used for session token signing

Flow:
  User login → Validate credentials → Create session → Store in KV
  → Return session cookie → On next request, validate cookie

4. FRONTEND ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━
Single-page application with:
- HTML: Static markup loaded once
- CSS: Obsidian-black dark theme, responsive design
- JavaScript: Modular scripts loaded dynamically

Entry point: frontend/index.html
Main app: frontend/scripts/main.js
  - Initializes UI on page load
  - Handles navigation between views (library, reader, settings)
  - DOM manipulation (safe methods, no innerHTML)

Reader implementations:
- EPUB: Text-only reader (frontend/scripts/text-only-reader.js)
  - Parses EPUB XML structure
  - Renders chapters as paginated text
  - Tracks reading progress
- PDF: PDF.js integration (frontend/scripts/pdf-reader.js)
  - Renders PDF pages
  - Navigation controls

5. API DESIGN
━━━━━━━━━━━
Cloudflare Pages Functions pattern:
- Each endpoint is a separate file
- Exports onRequestGet/Post/Put/Delete function
- Receives { request, env, next } context
- Returns Response object with JSON

Environment (env) includes:
- DB_1 through DB_10: Database bindings
- KV_SESSIONS: Session storage
- KV_CACHE: Query result caching
- Storage provider credentials
- JWT_SECRET

DATA FLOW EXAMPLES
═══════════════════════════════════════════════════════════════════════════════

EXAMPLE 1: UPLOAD A BOOK
━━━━━━━━━━━━━━━━━━━━━
1. User selects file and clicks upload in frontend (main.js)
2. FormData created with file, title, author, tags
3. POST /api/books/upload with FormData
4. Server (upload.ts):
   - Validates file (must be .epub or .pdf)
   - Creates DatabaseRouter from 10 D1 databases
   - Generates unique book ID: "book-1735001234-abc123"
   - Gets storage configs (10 providers in cascade order)
   - Uploads file to Google Drive (tries GDrive, then Dropbox if fails, etc.)
   - Stores metadata in DB_5 (hashed from book ID)
   - Returns book object with HTML snippet
5. Frontend receives response, updates UI with new book card

Database table structure:
```
books table (spreads across 10 databases by book_id):
  id, title, author, tags, cover_url, file_type, file_size,
  total_pages, uploaded_at, last_read_at, storage_provider, storage_id
```

EXAMPLE 2: LIST ALL BOOKS
━━━━━━━━━━━━━━━━━━━━━━
1. Frontend calls GET /api/books/list
2. Server (list.ts):
   - Checks KV_CACHE for cached results (5 minute TTL)
   - If cached, return immediately
   - If not cached:
     - Creates DatabaseRouter
     - Calls router.queryAll() which:
       - Iterates through all 10 databases (DB_1 to DB_10)
       - Executes SELECT query on each
       - Collects results from all 10 in array
       - Returns aggregated list
     - Maps results to add HTML snippets via generateBookCardHtml()
     - Caches result in KV for 5 minutes
     - Returns JSON response
3. Frontend renders each book with HTML snippet

EXAMPLE 3: READING A BOOK
━━━━━━━━━━━━━━━━━━━━━
1. User clicks book from library
2. Frontend calls GET /api/books/get?id=book-1735001234-abc123
3. Server (get.ts):
   - Creates DatabaseRouter
   - Gets correct database for this book ID (via hash)
   - Queries only that database for book metadata
   - Gets progress from same database (left join)
   - Gets reader settings from database
   - Gets storage configs for file download
   - Returns book, progress, settings, storage info
4. Frontend:
   - If EPUB: loads text-only-reader.js
   - If PDF: loads pdf-reader.js
   - Downloads file from storage provider
   - Renders content
5. As user reads:
   - Sends progress updates to POST /api/books/progress
   - Updates same database via router.queryForBook(bookId)

SECURITY MEASURES
═══════════════════════════════════════════════════════════════════════════════

1. XSS Prevention
   - NO innerHTML usage (safe: textContent, createElement, appendChild)
   - Properly escaped output
   - Content-Security-Policy headers (if configured)

2. Session Security
   - Session tokens stored in secure KV namespace
   - IP address validation (detect hijacking)
   - User-Agent validation (detect hijacking)
   - Automatic expiration
   - Secure cookie flags

3. API Authentication
   - Middleware validates session on every request
   - Protected routes require valid session token
   - Public routes: /, /index.html, /api/auth, /frontend/assets/

4. Database Access
   - No direct SQL injection (using D1 prepared statements with .bind())
   - Consistent hashing prevents guessing which database stores data
   - File IDs opaque to user

═══════════════════════════════════════════════════════════════════════════════
IMPORTANT FILES & THEIR ROLES
═══════════════════════════════════════════════════════════════════════════════

functions/_middleware.ts
- Validates session cookie on every request
- Checks IP and User-Agent for session hijacking
- Redirects unauthenticated users to login
- Returns 401 for API requests without valid session

functions/db-router.ts
- Core sharding logic using consistent hashing
- DatabaseRouter class manages 10 D1 instances
- createDatabaseRouter(env) initializes from DB_1-DB_10
- queryAll() for aggregate queries, queryForBook() for single queries

functions/storage-proxy.ts
- Abstracts 10 different storage providers
- Implements cascading failure logic
- uploadFile() tries providers in order
- downloadFile() retrieves from correct provider
- deleteFile() removes from storage

functions/api/books/list.ts
- Queries all 10 databases simultaneously
- Aggregates results
- Caches for 5 minutes in KV
- Generates HTML snippets for each book

functions/api/books/upload.ts
- Validates file type (EPUB or PDF only)
- Generates unique book ID
- Routes to correct database via hashing
- Uploads file with cascading storage fallback
- Stores metadata in correct database

functions/api/books/get.ts
- Routes to correct database by book ID
- Retrieves book, progress, and settings
- Returns file download info

frontend/index.html
- Single static HTML page
- Contains all layout and placeholders
- Loads CSS and JavaScript dynamically
- No server-side rendering

frontend/scripts/main.js
- Initializes app on page load
- Handles navigation between views
- Manages DOM updates
- Uses safe DOM methods (textContent, createElement)
- Calls API endpoints via fetch()

frontend/scripts/text-only-reader.js
- Parses EPUB format (ZIP with XML files)
- Extracts text from book chapters
- Renders as paginated text
- Tracks reading progress
- Sends progress updates to backend

DATABASE SCHEMA
═══════════════════════════════════════════════════════════════════════════════

(Replicated across all 10 D1 databases)

books
  id TEXT PRIMARY KEY
  title TEXT NOT NULL
  author TEXT
  tags TEXT (comma-separated)
  cover_url TEXT
  file_type TEXT ('epub' or 'pdf')
  file_size INTEGER
  total_pages INTEGER
  uploaded_at DATETIME
  last_read_at DATETIME
  storage_provider TEXT
  storage_id TEXT

progress
  id INTEGER PRIMARY KEY AUTOINCREMENT
  book_id TEXT FOREIGN KEY
  user_id TEXT
  percent INTEGER (0-100)
  current_page INTEGER
  current_chapter TEXT
  last_read_at DATETIME

shelves
  id TEXT PRIMARY KEY
  name TEXT NOT NULL
  description TEXT
  created_at DATETIME

shelf_items
  id INTEGER PRIMARY KEY AUTOINCREMENT
  shelf_id TEXT FOREIGN KEY
  book_id TEXT FOREIGN KEY
  added_at DATETIME

settings
  id INTEGER PRIMARY KEY (always 1)
  reader_theme TEXT (default: 'obsidian')
  reader_font_size INTEGER (default: 16)
  language TEXT
  notifications_enabled BOOLEAN

═══════════════════════════════════════════════════════════════════════════════
COMMON WORKFLOWS
═══════════════════════════════════════════════════════════════════════════════

ADDING A NEW BOOK ENDPOINT:
1. Create functions/api/books/[endpoint].ts
2. Define Env interface with DB_1-DB_10, KV namespaces, etc.
3. Export onRequestPost/Get/Put/Delete function
4. Create DatabaseRouter with createDatabaseRouter(env)
5. Use router.queryForBook(bookId) for single books
6. Use router.queryAll(sql) for listings
7. Return Response with JSON

ADDING A NEW STORAGE PROVIDER:
1. Edit functions/storage-proxy.ts
2. Add new StorageConfig type
3. Add provider to getStorageConfigs()
4. Implement upload/download/delete functions
5. Add credentials to environment variables

UNDERSTANDING DATA DISTRIBUTION:
1. Book ID gets hashed: hashBookId("book-123") → 5
2. Book stored in DB_5 (6th database, 0-indexed)
3. When querying: same hash → same database
4. List operations: query all 10, aggregate
5. Distribution roughly even if book IDs are random

TROUBLESHOOTING GUIDE
═══════════════════════════════════════════════════════════════════════════════

"Missing database binding: DB_5"
→ Configure all 10 databases in Cloudflare Pages settings

Book upload fails
→ Check storage provider credentials
→ Ensure at least one provider configured
→ Check file size limits

Reading progress not saving
→ Verify KV_CACHE is configured
→ Check database for progress table

Book not found after upload
→ Check correct database via hashBookId()
→ Verify schema initialized on that database
→ Check storage provider returned valid file ID

Session expires too quickly
→ Check JWT_SECRET is set
→ Verify KV_SESSIONS not full (quota limit)

PERFORMANCE CONSIDERATIONS
═══════════════════════════════════════════════════════════════════════════════

Caching Strategy:
- Book list cached for 5 minutes in KV
- Single book queries not cached (fresh data needed)
- Cache invalidated on upload/delete/update

Database Considerations:
- Each database 500MB max
- List operations hit all 10 databases (parallel)
- Single-book queries hit only 1 database (very fast)
- Writes always go to correct shard (no coordination needed)

Storage Considerations:
- First upload tries GDrive (typically 100-200ms)
- If fails, tries next provider (cascading)
- File ID stored to know which provider has file

Network:
- Files downloaded from storage provider directly (not proxied)
- Frontend downloads from GDrive/Dropbox/etc directly
